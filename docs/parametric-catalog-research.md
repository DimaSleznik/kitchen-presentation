# Архитектура data-driven параметрического каталога кухонных модулей

**Исследовательский документ для перехода от модели «один статичный GLB на SKU» к параметрическому, управляемому данными каталогу.**

Контекст: web-планировщик кухни (React + three.js / react-three-fiber). Потенциальный покупатель — Häcker Küchen (реальный немецкий производитель кухонь). Цель документа — обосновать гибридную архитектуру «шаблон + параметры + пресеты» ссылками на отраслевые стандарты и практики, а также наметить, где AI реально ускоряет наполнение и работу каталога.

Все ключевые факты снабжены ссылками на источники. То, что **не удалось подтвердить**, помечено явно (⚠️).

---

## 0. TL;DR для тех, кто принимает решение

- Вся профессиональная индустрия (2020/Cyncly, Compusoft Winner, Carat, KD Max, ArtiCAD, IKEA, OFML/pCon) моделирует шкаф как **параметрический шаблон + набор параметров + слой правил/валидации**, а не как фиксированный меш на каждый SKU. Наша гибридная идея — это и есть отраслевой мейнстрим, а не эксперимент.
- Данные строятся **слоями**: (1) геометрия, (2) коммерческие данные (цена/артикул/опции), (3) правила/ограничения. Эти слои связаны через идентичность артикула. Это прямой урок из OFML и IDM.
- **Важно про Häcker:** Häcker распространяет данные через немецкий стандарт мебельной отрасли **IDM (DCC)**, поставляемый в планировщики **CARAT и KPS** — **НЕ** через OFML/pCon (это стандарт офисной/контрактной мебели). Это меняет, на какой стандарт нам ориентироваться как на «родной» для Häcker. См. раздел 1.
- Декомпозиция корпуса (carcass) / фасадов (fronts) / фурнитуры (hardware) — это и есть европейская безрамная система; «система 32 мм» превращает размещение всех отверстий в арифметику. См. раздел 3.
- На вебе процедурная генерация корпуса + импортные GLB для фурнитуры/профилей + инстансинг + общий пул материалов — проверенный паттерн. Главный риск производительности — не полигоны, а **компиляция шейдеров** (наш известный фриз на переходах) и уникальные материалы. См. раздел 4.
- AI: LLM должен быть **«предлагателем параметров», а не генератором геометрии** — через strict tool calling / structured outputs. Картинки для каталога — рендер depth/normal из three.js → ControlNet на fal.ai. См. раздел 5.

## 1. OFML / pCon / EasternGraphics — и что на самом деле использует Häcker

### 1.1 Что такое OFML

**OFML** (Office Furniture Modeling Language) — свободно доступный, вендор-нейтральный стандарт описания, конфигурирования и презентации сложных конфигурируемых продуктов. Возник в офисной мебели (1998), но применяется и в других секторах.

- Создан в 1998 по заказу немецкого отраслевого союза **BSO → ныне IBA** (Industrieverband Büro und Arbeitswelt). Технический разработчик и держатель инструментария — **EasternGraphics GmbH** (Ильменау). Спецификации принадлежат IBA и доступны бесплатно. ([Wikipedia: OFML](https://en.wikipedia.org/wiki/OFML), [IBA specifications](https://iba.online/knowledge/en/raeume-planen/ofml/specifications/), [IBA glossary](https://iba.online/knowledge/en/raeume-planen/ofml/glossary/))
- Задача OFML: единое платформенно-независимое описание продукта с большим числом свойств, объединяющее в одной модели **логику планирования, графику/геометрию, коммерческие данные и данные конфигурации**.

### 1.2 Слоистая модель OFML (ключевой урок)

OFML намеренно разбит на отдельные части-подспецификации («OFML Parts»). Это самый важный структурный вывод. ([IBA specifications](https://iba.online/knowledge/en/raeume-planen/ofml/specifications/))

| Часть | Имя | Слой / роль |
|------|------|-------------|
| Part I | **ODB** — OFML Database | **Геометрия.** Табличное описание иерархической 2D/3D-геометрии, точек привязки/соединения, функциональных таблиц. |
| Part III | **OFML** (ядро) | Объектно-ориентированное ядро/рантайм: статьи (articles), свойства, геометрические примитивы (`OiBlock`, `OiCylinder`, `OiSweep`, `OiHole`…), правила. |
| Part IV | **OCD** — OFML Commercial Data | **Коммерческий слой.** Артикулы, цены, variant-коды, логика опций, надбавки, описания/длинные тексты для КП и заказов. |
| Part VI | **OAM** — OFML Article Mappings | **Слой связывания** коммерческих свойств/variant-кодов с параметрами геометрии. |
| Part VII | **OEX** — Business Data Exchange | XML-обмен заказами (Orders, Order Responses, Order Changes). |

Источники: [OFML core spec PDF (2.0r3)](https://iba.online/site/assets/files/24390/ofml_20r3-en.pdf), [Article/CompositeArticle interface spec](https://docs.pcon-solutions.com/ofml-specs/OFML/article_interface_1.4_en.pdf), [OCD Part IV spec](https://www.yumpu.com/en/document/view/40260319/specification-ocd-ofml-commercial-data-ofml-part-iv-version-30), [OAM Part VI spec](https://iba.online/knowledge/site/assets/files/6210/oam_1_0-en.pdf).

**Как устроены параметрические данные:**
- **Геометрия (ODB / ядро OFML):** описывается *параметрически*, не как запечённый меш. Примитивы (`OiBlock`, `OiSweep`, `OiHole`, …) управляются значениями свойств.
- **Коммерческий слой (OCD):** ключевой механизм — **variant code**: значения конфигурируемых характеристик *сериализуются в номер артикула*. Один параметрический article ⇒ множество продаваемых SKU, которые **генерируются, а не перечисляются**. Цена = базовая + зависящие от свойств надбавки/соотношения, а не плоский прайс-лист.
- **Правила/ограничения:** restrictions на интерфейсе Article (бизнес-правила допустимых комбинаций), плюс рантайм-проверка валидности и **динамическая фильтрация доступных вариантов** по текущему выбору. ([Article interface spec](https://docs.pcon-solutions.com/ofml-specs/OFML/article_interface_1.4_en.pdf))
- **Сборки:** `CompositeArticle` — сборка из под-артикулов (родитель + дети, синхронизируемые). Прямой аналог кухонного ряда: ряд — родитель, шкафы/фасады/столешницы/техника — дети со своими свойствами и общими ограничениями.

### 1.3 Экосистема pCon (кратко)

- **pCon.planner** — бесплатный 3D-планировщик помещения (Pro-версия грузит B2B OFML-каталоги). ([pCon.planner](https://pcon-solutions.com/planner/))
- **pCon.catalog** — онлайн-каталог/конфигуратор-портал. ([pCon.catalog](https://pcon-solutions.com/pcon-catalog/))
- **pCon.basket** — коммерческое приложение (КП/заказы).
- **pCon.update** — дистрибуция версионированных OFML-датасетов производителей дилерам.
- Авторинг данных: **pCon.creator** (профессиональный), **easyOFML Creator** (упрощённый, с импортом из Excel/BMECat). ([pCon.creator manual](https://docs.pcon-solutions.com/pCon/creator/release/2.22.1/pCon.creator_2.22.1_Manual_EN.pdf), [easyOFML Creator](https://docs.pcon-solutions.com/pCon/plannerPlugins/easyOFML/Dokumentation_easyOFML_creator_Plugin_EN.pdf))

### 1.4 Что РЕАЛЬНО использует Häcker (важная поправка) — ПОДТВЕРЖДЕНО

Распространённое предположение, что «европейский производитель = OFML/pCon», **в случае Häcker неверно**. Немецкая *кухонная* отрасль (в отличие от офисной/контрактной) стандартизирована на ином стеке:

- **Häcker подтверждённо использует CARAT.** Häcker явно присутствует в официальном списке производителей CARAT productinfo (наряду с nobilia, Nolte, Schüller, Leicht…). CARAT распространяет дилерам «данные, поставляемые производителем». ([CARAT productinfo Herstellerübersicht](https://doc.caratcloud.com/carat/onlinehelp/de/Content/4_PlanungKauf/400700/CARAT_productinfo_%20Herstelleruebersicht.htm), [CARAT productinfo](https://www.carat.de/produkte/carat-productinfo))
- **Базовый стандарт данных — IDM (Integriertes Datenmodell) от DCC.** IDM — это «кухонный формат» немецкой мебельной отрасли для планировочно-сложных продуктов; разработан DCC совместно с SHD. **Häcker — участник DCC в категории кухня/ванная**; на базе IDM-данных производится ~6000 каталогов в год. Производители выкладывают IDM-данные авторизованным софт-хаусам (CARAT, KPS), которые собирают и распространяют каталоги дилерам. ([MÖBELMARKT: SHD и DCC — IDM для всех подотраслей](https://www.moebelmarkt.de/beitrag/shd-und-dcc-integriertes-datenmodell-idm-fuer-alle-teilbranchen), [DCC Cat@web FAQ](https://www.catatweb.de/index.php?id=12), [Küchenplaner-Magazin: все данные Häcker «в один клик» через Product Pilot](https://www.kuechenplaner-magazin.de/themen/detail/news/alle-haecker-daten-auf-einen-klick/))
- **IDM — параметрический по схеме.** Каждая статья — `ITEM` с `TYPE_NO`; геометрия — `CARCASE_BASIC_SHAPE` (параметризованная графика) с `BASIC_SHAPE_PARAMETERS` (b/h/t = ширина/высота/глубина + номинал и допустимый диапазон `BASIC_SHAPE_FROM`/`BASIC_SHAPE_TO`); сторона навески — `CONSTRUCTION_ID` (N/L/R/J); варианты — `FEATURE_NO`+`OPTION_NO` с `RESTRICTIONS`. Цена — через `PRICE_DEFINITION` + условные правила. ([IDM 2.8.1 XML Schema](https://www.dcc-moebel.org/dochtml/IDM_2_8_1_XML_Schema/output.toc.htm))
- **KPS / KPS.MAX (SHD)** — второй из двух доминирующих планировщиков немецкого кухонного ритейла. Прямого источника, называющего Häcker, найти не удалось, но крупные производители обычно поставляют данные под оба, а Häcker «Product Pilot» описан вендор-нейтрально. Статус: **ВЕРОЯТНО**, не подтверждено документально. ([KPS.MAX](https://www.kps-max.com/))

**Вывод по Häcker (уверенность высокая):** «родной» стандарт данных Häcker — **IDM (DCC), доставляемый через CARAT/KPS**. OFML/pCon — **нет данных** (это стандарт офисной мебели). Для нас это значит: если когда-либо встанет вопрос прямого импорта реальных данных Häcker, целевой формат — **IDM**, а не OFML. Но как *референс архитектуры* OFML и IDM устроены почти одинаково (слой геометрии + коммерческий слой + правила), и оба подтверждают нашу слоистую модель.

### 1.5 Почему это важно для нашей архитектуры

- **Разделяй слои, как OFML/IDM:** (1) параметрическая геометрия, (2) коммерческие данные (цена/SKU/опции), (3) правила валидности/зависимостей. Это ядро урока.
- **Шаблон = параметрический article, а не фиксированный меш.** Геометрия генерируется из примитивов, управляемых параметрами.
- **SKU генерируются из конфигурации, а не перечисляются** (механизм variant code). Наши «пресеты» — это именованные канонические наборы значений свойств внутри пространства вариантов.
- **Правила — первоклассная сущность.** Restrictions + рантайм-валидация + динамическая фильтрация выбора. Пресет = заранее провалидированная точка в пространстве правил.
- **Кухонный ряд = composite/сборка** (родитель + дети, синхронизация общих ограничений: высоты, выравнивание, непрерывность столешницы).
- **Цена тоже параметрическая:** база + надбавки по свойствам, а не плоский прайс.

⚠️ **Не подтверждено:** названия «GoDigital / GoOFML / DataConnector» как компоненты OFML (вероятно, маркетинговые имена продуктов); существование отдельной публичной спецификации «OAS» и «Part V»; гарантированно нет публичного OFML-датасета Häcker (портал pCon — некроулящийся SPA, дилерский pCon.update под логином — 100% исключить нельзя, но публичных подтверждений нет).

## 2. Как профессиональные планировщики кухонь моделируют шкафы

Сквозной вывод: **все** инструменты используют «параметрический шаблон + параметры + слой правил», а данные конкретного производителя **подгружаются отдельно** от движка рендера/планирования. Никто не поставляет фиксированный меш на каждый покупаемый SKU.

### 2.1 Как представлен «шкаф»

- **Compusoft Winner / Winner Flex (Cyncly)** — самая чёткая формулировка: у каждого изделия есть «3D-параметрическая модель + метаданные, превращающие простой 3D-объём в цифрового двойника физического изделия». В UI редактируемые параметры (например, переключить 2 ящика → 3, добавить выкатные). ([Cyncly: AI-трансформация дизайна кухни](https://www.cyncly.com/resources/blog/cyncly-ai-transformation-of-kitchen-design-and-sales), [Winner support](https://winnerdesign.support.compusoftgroup.com/hc/en-gb/articles/12112948122001-How-can-I-show-internal-cabinetry-of-units))
- **2020 Design / Design Flex (Cyncly)** — изделия конфигурируемы и размерно-управляемы. В большинстве каталогов есть **«User Code», пересчитываемый при ресайзе** (W/H/D), и стабильный **«Manufacture Code»** для заказа. Терминология 2020 — «configurable» + «rules-based validation». ([Saberis: коды каталога 2020](https://www.saberis.com/Module/Kitchen), [2020 Design Flex](https://www.2020spaces.com/2020products/designflex/2020-design-manufacturer-content/))
- **Carat (MHK Group)** — не геометрическое ядро, а каталого-управляемый планировщик, рендерящий немецкие данные **IDM** (см. 1.4). ([carat.de](https://www.carat.de/))
- **KD Max (YFCAD)** — «все шкафы, двери, ручки параметрические». Важно: (а) шаблон живёт на уровне глобального каталога, и правки **не распространяются** на уже размещённые экземпляры (нужно переприменять); (б) сложные случаи (угловые 45°) **запекаются в статичную нередактируемую геометрию**. ([KD Max about](https://cabinetsbycomputer.com.au/about-kd-max/), [V10 45° KB](https://support.cabinetsbycomputer.com.au/support/solutions/articles/60001566971-kd-max-v10-45-degree-cabinets))
- **ArtiCAD (UK)** — шкафы = «units», редактируются через Customise (производитель/линейка, тип двери/ящика, ручка+позиция). Каталожный билдер ArtiCUT описан как «параметрический». ⚠️ Публичные страницы называют параметрическим именно *каталожный слой*, а не размещённый рантайм-юнит. ([ArtiCUT](https://www.articad.com/software/articut/))

### 2.2 Структура каталога — ортогональные оси

Универсальная модель: **независимые выбираемые оси поверх корпуса**, а не N×M запечённых вариантов. Поток выбора почти везде один: **линейка/стиль фасада → шкаф → отделка/цвет → ручка → фурнитура**; техника — отдельная категория.

- **Winner:** выбор «Model» (= стиль фасада/линейка) глобально или по юниту → отделки из отдельных списков → ручка вложенным дропдауном. ([смена стиля фасада](https://winnerdesign.support.compusoftgroup.com/hc/en-gb/articles/360009987997-Changing-or-editing-a-door-style))
- **ArtiCAD** (самый явный поток): Change Manufacturer → Change Range → Customise Door → handle + позиция → цвет/отделка; цвет фасада независим от корпуса/цоколя.
- **KD Max:** раздельные библиотеки — Cabinets, Doors, Carcasses, Accessories, Hardware, Appliances, Decoration; отделки — drag&drop сэмплов. ([yfcad.sg](https://www.yfcad.sg/yfcad/kd-max/))
- **Carat/IDM:** домены `CATALOG · SUPPLIER · SERIES (линейка→группа→ITEM) · FEATURE_DEFINITION (варианты/опции/ограничения) · PRICE_DEFINITION`.

**Нюанс (подтверждён для IDM, для остальных — вывод):** для корпусной мебели корпус/фасад/декор/ручка чаще являются *вариантными измерениями (features/options) внутри одного датасета производителя*, а не физически отдельными базами. Самая отделимая ось — **техника** (отдельные прайс-источники). Для нашей архитектуры это всё равно ложится на ортогональные слои пресетов — просто помним, что «по-взрослому» это часто feature-опции на одном изделии, а не три join-таблицы.

### 2.3 Связь SKU / прайс-артикулов с моделью

Универсальный паттерн: **стабильный артикул производителя + ценовой слой, привязанный к конфигурации**, часто плюс второй производный/отображаемый код.

- **2020:** User Code (пересчёт при ресайзе) + Manufacture Code (заказ); прайс на «price level», дилер применяет скидку/множитель; **Global Style Validation (GSV)** проверяет валидность комбинаций при отправке заказа.
- **Winner:** артикулы + обязательные доп-коды; цена через **Cost Index (закупка)** и **Sales Index (продажа)**, требующие утверждения.
- **Carat/IDM:** цена из вариантов/размеров через `PRICE_DEFINITION`; заказ через EDI; каталоги лицензируются/раздаются через **Cat@web**.
- **KD Max — исключение:** прайс-листов по SKU нет; расчёт = площадь панелей × своя стоимость материалов (make-to-order). ([KD Max features](https://cabinetsbycomputer.com.au/kd-max-features/))

### 2.4 IKEA METOD — самый чистый публичный пример декомпозиции

METOD — учебниковый пример отделения «коробки» от «обёртки», и планировщик IKEA имеет это предзапрограммированным.

- **Каркасы (frames)** — параметрическая база, продаются отдельно в стандартных размерах: база **высота 80 см, глубина 60 см (опция 37 см), 18 мм**, дискретные ширины (40/60/80 см и др.). Страница каркаса прямо гласит: «Двери, ящики, столешница, полки, петли, цоколи и накладные панели продаются отдельно». ([METOD base 80×60×80](https://www.ikea.com/gb/en/p/metod-base-cabinet-frame-white-10205633/), [METOD cabinets PDF](https://www.ikea.com/gb/en/files/pdf/a1/5c/a15c8aab/ikea-metod-cabinets-and-interior-fittings.pdf))
- **Оси (каждая — отдельная покупка):** Frame (база/верх/высокий/угол) × Fronts (двери/фасады ящиков, стили/отделки) × Hinges (UTRUSTA: 110°/153°/45° по сценарию) × Drawers/interiors (MAXIMERA) × Handles × Shelves/Legs/Plinth/Cover panels.
- Планировщик IKEA имеет «все размеры METOD и положения отверстий петель предзапрограммированными». Это 1:1 наша модель: **template (генератор геометрии каркаса) + parameters (размер, сторона навески) + presets (стиль фасада, отделка, линейка фурнитуры)**. ([nakedkitchens: IKEA hack](https://www.nakedkitchens.com/blog/ikea-kitchen-hack-bespoke-fronts))

### 2.5 Поправки терминологии (⚠️)

- «2020 Cap» — это софт офисного пространственного планирования, **не** инструмент авторинга каталога (авторинг = Catalog Tools / Content Studio).
- «Carat by KPS, под Cyncly» — **неверно**: Carat принадлежит MHK Group; связь Cyncly — интеграционная, не владение. KPS — отдельный вендор.
- «Параметрический» vs «configurable»: Winner/Cyncly говорят «parametric model… digital twin»; 2020 — «configurable»; IDM/OFML — параметрические на уровне схемы; ArtiCAD называет параметрическим только каталожный билдер.

### 2.6 Выводы для архитектуры

1. **Template + parameters + rules — универсальная модель, не фиксированные меши.** Копировать как референс схемы стоит IDM (`CARCASE_BASIC_SHAPE` + `BASIC_SHAPE_PARAMETERS` FROM/TO + `CONSTRUCTION_ID` + `FEATURE/OPTION`) и OFML/OCD.
2. **Стиль фасада/линейка, отделка/цвет и фурнитура — ортогональные слои пресетов поверх корпуса** (блюпринт — IKEA METOD и ArtiCAD). Отделку применять *к* геометрии, не запекать. Не плодить N×M.
3. **Два идентификатора + ценовой слой по конфигурации:** стабильный артикул (заказ) + производный/отображаемый код (пересчёт при ресайзе). Прайс держать подключаемым: и SKU-привязанные прайс-листы (перепродажа), и расчёт «площадь×материал» (make-to-order).
4. **Контент каталога — загружаемые данные, полностью отделённые от движка.** Геометрический слой отдельно от коммерческого; связь — через идентичность артикула.
5. **Запланировать явный fallback «параметрика → статика»** (как KD Max запекает угловые 45°) и сознательно решить политику распространения правок на размещённые экземпляры (у KD Max — НЕ распространяются).

## 3. Стандартная декомпозиция и параметры конструкции кухонного шкафа

Определяющий принцип европейской (безрамной, full-access) системы — **разделение слоёв**: корпус, фасады и фурнитура независимы и комбинируются свободно. Это ровно то, что нужно параметрическому генератору.

### 3.1 Безрамная («европейская») vs face-frame («американская»)

- **Безрамная:** нет передней рамы; двери/ящики *и есть* фасад. Боковины/верх/низ/полки — обычно **~18–19 мм** ЛДСП; задняя стенка тонкая (**~6 мм**). Петли крепятся **прямо к толстой боковине**. Соединения — шканты, конфирматы, эксцентрики. Даёт до **~15%** больше полезного объёма. ([Conestoga](https://www.conestogawood.com/frameless-vs-framed-cabinet-guide/), [Dura Supreme](https://www.durasupreme.com/framed-vs-frameless-cabinetry/))
- **Face-frame:** массивная рама спереди; петли/двери привязаны к раме. (Для нас актуальна безрамная — европейский рынок, Häcker.)

**Почему важно:** в безрамной системе контур корпуса определяется только толщиной панелей и габаритами — нет геометрии рамы. Слой фасадов полностью отвязан → один корпус «перешивается» любым фасадом сменой параметров.

### 3.2 Слои

| Слой | Компоненты | Параметрические заметки |
|---|---|---|
| **Корпус (carcass)** | 2 боковины, верх (или царги у баз), низ, задняя стенка, полки | W×H×D + толщина панели. Боковина — несущая и опорная для петель. |
| **Фасады (fronts)** | Двери, фасады ящиков | Отдельный слой; взаимозаменяемы; стиль/цвет/материал. Размеры кратны 32 мм минус зазор. |
| **Фурнитура (hardware)** | Петли (чашка 35 мм), направляющие ящиков, ручки, регулируемые ноги + клипсуемый цоколь, стяжки | Привязка к сетке системных отверстий. |
| **Внутреннее наполнение** | Полки (на полкодержателях в системных отверстиях), выкатные, лотки, угловые решения | Положение полок снапится к рядам сетки 32 мм. |

**Опорные числа фурнитуры (важны для параметрики):**
- **Скрытая европетля:** чашка **Ø35 мм**, глубина врезки **~12–13 мм**, центр чашки **~23 мм** от края двери, «boring distance» **~3–6 мм** (регулирует накладку). Ответная планка на корпусе — **~37 мм** от переднего края, два отверстия **32 мм** друг от друга. ([WoodWeb: Blum drilling](https://woodweb.com/knowledge_base/Blum_hinge_drilling_locations.html), [Rockler](https://www.rockler.com/learn/installation-tips-european-hinges))
- **Hettich Sensys** (альтернатива Blum): паттерн чашки **52 × 5.5 мм**, толщина двери 15–32 мм. ([Hettich eShop](https://shop.hettich.com/au_EN/Hinges/Concealed-hinges/Sensys-fast-assembly-concealed-hinges/))
- **Направляющие** (Blum TANDEM/MOVENTO/LEGRABOX, Hettich): вертикальный монтаж кратен **32 мм**; при этом **длины** направляющих Blum идут по дюймовой сетке (3″≈76 мм) — моделировать как два разных параметра. ([Blum TANDEM](https://www.blum.com/us/en/products/runnersystems/tandem/programme/))

### 3.3 «Система 32 мм» / системные отверстия — движок размещения

- Вертикальные ряды отверстий **Ø5 мм с шагом 32 мм** (line boring) в боковинах. ([Wikipedia: 32 mm system](https://en.wikipedia.org/wiki/32_mm_cabinetmaking_system), [True Position Tools](https://truepositiontools.com/learn/how-to/getting-to-know-the-32-mm-system-for-cabinet-making/))
- **Передний ряд = 37 мм** от переднего края; задний ряд — точное кратное 32 мм от переднего (оба на одной решётке). Глубина отверстия **~12–14 мм**. ([davelers 32mm Basics](https://davelers.com/32mm/Basics/157.html))
- **Системные отверстия Ø5 мм** (полки/петли/направляющие) vs **конструкционные/присадочные Ø8 мм** (шканты) — это **разные паттерны**.
- **Почему 32 мм:** историко-механическая причина — это было ближайшее кратное, на котором свёрла садились в линию на присадочном станке тех лет. Затем стало гармонизирующим стандартом (и через IKEA — глобальным).
- **Как стандартизирует:** ответные планки петель (2 отверстия на 32 мм), направляющие (кратно 32 мм), полкодержатели (любое системное отверстие). Высоты фасадов кратны **32 мм минус зазор**, чтобы фурнитура была симметрична сверху/снизу при любой высоте.

**Почему важно:** сетка превращает размещение отверстий в чистую арифметику. Генератор ставит передний ряд на 37 мм, снапит всё остальное на решётку `37 + 32·n`, ограничивает высоты фасадов `32·k − зазор`. Симметрия сверху-вниз → одни и те же офсеты фурнитуры для любой высоты.

### 3.4 Размерные сетки европейских кухонь

- **Ширины (база и верх делят сетку):** 150, 200, 300, 400, 450, 500, 600, 800, 900, 1000, 1200 мм — шаг **100 мм** (полушаг 50 мм внизу/для доборов). IKEA METOD — грубее (200/400/600/800). ([Arcadium3D](https://arcadium3d.com/articles/standard-kitchen-cabinet-sizes-eu-uk-us), [Better Kitchens](https://www.betterkitchens.co.uk/blog/kitchen-buying-tips-and-advice-4/kitchen-cabinet-dimensions-guide-for-standard-upper-cabinets-75))
- **Высота базы / высота столешницы — конкурирующие стандарты (важно):** по референсу Nobilia сосуществуют:

| Стандарт | Корпус | Цоколь | Высота столешницы (без столешн.) |
|---|---|---|---|
| Standard / UK | 720 мм | 150 мм | 870 мм |
| German «XL» | 792 мм | 100 мм | 892 мм |
| «Maxi» | 864 мм | 70 мм | 934 мм |

  IKEA METOD: 80 см каркас + 8 см ноги = 88 см. ([German Kitchen Specialist — Nobilia](https://www.germankitchenspecialist.co.uk//nobilia-kitchen-carcass-construction-heights))
- **Глубина базы:** корпус **~560–580 мм**; столешница **~600–650 мм**. METOD база = 600 мм (опция 370 мм).
- **Верхние шкафы:** высоты **360/575/720/900 мм**; глубина **~300–370 мм**; над столешницей **~450–600 мм**.
- **Высокие/пеналы:** **~1825/1950/2150 мм** (UK) или 2000/2200 мм (METOD); глубина = базе.
- **Цоколь:** **100–150 мм** (немецкие — к 100, UK — к 150), через регулируемые ноги + отдельная клипсуемая планка. Столешница **~20–40 мм**.
- **Толщина панели:** **18 мм** ЛДСП по умолчанию (16/19 мм — варианты); это **главный вычитаемый член**: внутр. ширина = внешняя − 2×толщина. Безопасный пролёт полки 18 мм ~600–800 мм (ограничивает ширину до серединной опоры). Задник тонкий (~6 мм).

**Почему важно:** высота столешницы — это сборка `корпус + цоколь + столешница`, каждый член — отдельный параметр, а не хардкод-константа.

### 3.5 Выводы для генератора

- **Три независимых слоя** — корпус, фасады, фурнитура.
- **Сетка 32 мм — движок размещения:** передний ряд 37 мм; всё на `37 + 32·n`; Ø5 vs Ø8 — разные паттерны.
- **Высоты фасадов = `32·k − зазор`.**
- **Сетка ширин** 100 мм (50 мм внизу): 150…1200 мм; пролёт полки ≤ ~600–800 мм (провисание 18 мм).
- **Высота столешницы = корпус + цоколь + столешница**, все параметры: 720+150=870 (UK), 792+100=892 (German XL), 864+70=934 (Maxi). Цифру «780 мм» (Arcadium3D) считать ненадёжной.
- **Толщина 18 мм по умолчанию** — мастер-офсет для всех внутренних размеров и присадки.

⚠️ **Не подтверждено:** конкретный номер DIN/EN на эти размеры (система 32 мм — де-факто отраслевая, а не одна цитируемая DIN-норма); «780 мм» противоречит точным значениям Nobilia.

## 4. Процедурная / параметрическая геометрия на вебе (three.js)

**Вердикт для архитектуры:** гибрид — то, к чему сходятся реальные конфигураторы: **процедурно генерировать корпус** (боковины/верх/низ/задник/полки как параметризованные box-панели), **импортировать GLB для фурнитуры и профилированных фасадов** (ручки, петли, профильные двери), **инстансить повторяющиеся части**, **агрессивно шарить материалы**, **прекомпилировать шейдеры** (закрывает наш фриз на переходах).

### 4.1 Параметрическая генерация меша в рантайме

- **Композиция box-панелей (проще всего, идеально для корпуса):** каждая панель — `THREE.BoxGeometry(w,h,d)`, позиционирование через `translate()`/группы. Так сделан процедурный кухонный дизайнер на Codrops: шкафы/столешницы/верхние — `BoxGeometry`, позиционируются накоплением X-офсета, реальные константы, алгоритм деления ширины со стандартными ширинами + масштабируемый добор. ([Codrops, 2025: процедурный кухонный дизайнер на three.js](https://tympanus.net/codrops/2025/07/29/exploring-the-process-of-building-a-procedural-3d-kitchen-designer-with-three-js/))
- **2D-shape → extrude (для скруглений/профилей/CNC):** `Shape` + `ExtrudeGeometry`. Конфигуратор Salsita/KILO построил TypeScript-DSL 2D-форм, из которых один код делает и 3D-вьюпорт, и SVG-превью, и **DXF для ЧПУ**; DSL работает и в Node, и в браузере → «что рендерится, то и производится». ([Salsita: параметрический 3D-конфигуратор](https://salsita.ai/blog/implementing-a-challenging-product-configurator-with-3d-parametric-models/))
- **Ручной `BufferGeometry`** — низший уровень, для корпуса избыточен.
- Паттерн `Board`-компонента (M-Click): «каждая часть мебели — Board, знающий свои размеры и позицию относительно родителя», Zustand держит вложенное состояние секций/полок, шаг полок пересчитывается на каждое изменение с учётом 16/18 мм. ([DEV.to / M-Click](https://dev.to/panda_koder/building-a-3d-furniture-configurator-from-scratch-a-solo-developers-journey-with-react-threejs-3l6a))

### 4.2 Производительность для многих шкафов в сцене

Бенчмарк-ориентир: **< 100 draw calls/кадр**. ([utsubo: 100 tips](https://www.utsubo.com/blog/threejs-best-practices-100-tips))

- **`InstancedMesh(geometry, material, count)`** — одна геометрия, много трансформов, **один draw call**. `setMatrixAt()` (+`instanceMatrix.needsUpdate`), `setColorAt()`. Идеально для идентичных повторов: ручки, петли, направляющие, полкодержатели, детализация системных отверстий. Быстрее BatchedMesh до ~100k. ([Instanced vs Batched](https://discourse.threejs.org/t/how-to-choose-between-instancedmesh-and-batchedmesh/81221))
- **`BatchedMesh` (r156+)** — разные геометрии при общем материале, один draw call, `setVisibleAt()`; медленнее истинного инстансинга (~1.5–2×). Правило: одна геометрия → InstancedMesh; смешанные при общем материале → BatchedMesh.
- **`BufferGeometryUtils.mergeGeometries()`** — запечь N статичных панелей с общим материалом в одну геометрию (N → 1 draw call). Минус: теряются индивидуальные трансформы/raycasting/смена материала. Хорошо для *финализированного* шкафа, плохо во время перетаскивания слайдеров. ([BufferGeometryUtils](https://threejs.org/docs/pages/module-BufferGeometryUtils.html))
- **В r3f/drei:** `<Instances>`/`<Instance>`, `<Merged>`; `gltfjsx model.glb --instanceall` авто-генерит `useGLTF` + `Merged`. Скиннинг-меши инстансинг **не** поддерживают. ([Codrops Instances](https://tympanus.net/codrops/2025/07/10/three-js-instances-rendering-multiple-objects-simultaneously/), [gltfjsx](https://github.com/pmndrs/gltfjsx))
- **Шаринг материалов — сам по себе рычаг:** three.js батчит меши с *идентичными* материалами; новый материал на панель убивает батчинг. Кэшировать текстуры в `Map` по URL, атласить.

### 4.3 Процедурная геометрия vs GLB на SKU — компромиссы

- **Трафик/размер:** одно параметрическое определение + маленькая общая GLB-библиотека фурнитуры решительно бьёт N GLB на SKU. Для поставляемых GLB — сжатие **Draco** (геометрия −90–95%), **KTX2/Basis** (текстуры, ~×10 VRAM), Meshopt. Пайплайн: `gltf-transform optimize ... --compress draco --texture-compress ktx2`.
- **Память:** 200 КБ PNG → 20+ МБ VRAM; формат текстур важнее полигонов. **three.js не собирает GPU-ресурсы сам** — при смене конфигурации обязательно `geometry.dispose()`/`material.dispose()`/`texture.dispose()`, следить за `renderer.info.memory`. Критично для конфигуратора, пересобирающего геометрию на каждый слайдер.
- **Что трудно процедурно (→ держать как GLB):** фаски/скругления, профильные/рамочные фасады, декоративные молдинги; реалистичная фурнитура (ручки/петли/ноги — смоделировать раз, импортировать, инстансить); UV древесного волокна.
- **Гибрид (подтверждён по кейсам):** процедурный/плейсхолдерный корпус + размещение GLB по параметрам. Codrops буквально рисует цветные плейсхолдер-боксы (`MeshBasicMaterial`), затем async-подменяет реальные GLB. Наш «template (процедурный корпус) + presets (GLB-фурнитура/фасады по параметрам) + инстансинг» — та же форма.
- **UV древесины (главный провал по качеству):** box-UV растягивает текстуру по граням → плохо для панелей разного размера. Лучшее решение от мейнтейнеров three.js — **сэмплировать по мировой позиции вершины (triplanar)**, «как будто вырезано из бесконечного куска дерева», убирает растяжение при любом размере. ⚠️ В ядре three.js нет встроенного triplanar — это шейдер/TSL-аддон или `onBeforeCompile`-патч. ([three.js discourse: procedural furniture](https://discourse.threejs.org/t/procedural-parametric-furniture-in-threejs/68760))
- **Компиляция шейдеров (наш фриз на переходах):** каждая уникальная комбинация material+текстуры+lights+shadows компилирует **отдельную GPU-программу**; первый рендер новой программы = видимый фриз. Полевые данные: сцена ~20 мешей / ~30 уникальных материалов + тени компилировалась **~3.5 с**, упала до **~0.85 с** за счёт (а) `MeshPhysicalMaterial`→`MeshStandardMaterial`, (б) отказа от `transmission` (удваивал время), (в) **плейсхолдер-текстур вместо `null`** (общие шейдер-константы). Firefox ~×5 медленнее Chrome. Митигации:
  - **Прекомпиляция:** `renderer.compile(scene, camera)` или лучше `renderer.compileAsync(...)` (использует `KHR_parallel_shader_compile`, компилирует вне main thread, возвращает Promise). ⚠️ Поддержка compileAsync неравномерна по браузерам/драйверам — best-effort. ([shader-compile thread](https://discourse.threejs.org/t/reducing-shader-compile-time-on-scene-initialization/56572), [WebGLRenderer.compile](https://threejs.org/docs/#api/en/renderers/WebGLRenderer.compile))
  - **Минимизировать перестановки материалов** — фиксированная палитра общих `MeshStandardMaterial` (параметр → материал по словарю): меньше программ + лучше батчинг. Самый высокорычажный пункт.

> ⚠️ Прямая связь с нашим внутренним мемо `shader-compile-freeze-transitions`: корень тот же — синхронная компиляция WebGL-программ. Прекомпиляция/прогрев + малая палитра материалов — основное лекарство.

### 4.4 CSG для отверстий и вырезов

- **`three-bvh-csg`** (gkjohnson) — лучший на сегодня CSG (на `three-mesh-bvh`, **>100× быстрее** BSP). API: `Brush` + `Evaluator().evaluate(a, b, SUBTRACTION|HOLLOW|…)`. Для вырезов **мойки/варочной/кабельных/отверстий петель** — `SUBTRACTION`/`HOLLOW`; резак (Brush B) должен быть **водонепроницаемым 2-manifold**. ⚠️ UV на результатах документированы слабо. ([repo](https://github.com/gkjohnson/three-bvh-csg))
- **`@react-three/csg`** — декларативная r3f-обёртка (`<Geometry><Base/><Subtraction/></Geometry>`). ⚠️ README помечен WIP.
- **Практика:** **не делать CSG на каждый тик слайдера** — резать один раз при коммите выреза, кэшировать, по возможности в Web Worker; чисто визуальные круглые отверстия часто дешевле имитировать alpha-маской.

### 4.5 Главные практические выводы

- **Гибрид — проверенный паттерн:** процедурный box/extrude-корпус + GLB-пресеты фурнитуры/профильных фасадов, размещаемые по параметрам, async-подмена за плейсхолдер-боксами.
- **Корпус = параметризованные `BoxGeometry`-панели**; `ExtrudeGeometry` только для фасок/профилей/CNC; ручной `BufferGeometry` почти не нужен.
- **Draw calls:** InstancedMesh для идентичных повторов; BatchedMesh для смешанных при общем материале; mergeGeometries для финализированного шкафа. Цель < 100/кадр.
- **Материалы — реальная цена, не полигоны.** Малая фиксированная палитра `MeshStandardMaterial`, плейсхолдер-текстуры вместо `null`, атлас/кэш.
- **Прогрев шейдеров** (`compile`/`compileAsync`) перед показом новой геометрии — прямое лекарство от нашего фриза.
- **Dispose** генерируемой геометрии/материалов/текстур на каждой пересборке; GLB-фурнитуру — Draco+KTX2.
- **Вырезы — `three-bvh-csg`** (`SUBTRACTION`/`HOLLOW`), резать один раз на коммите, кэшировать.
- **Растяжение древесины — triplanar/мировая позиция**, не box-UV.

## 5. AI / нейросетевой угол (2024–2026)

### 5.1 LLM как «предлагатель параметров» через structured outputs / tool calling

Главное изменение с 2024 — оба крупных провайдера **гарантируют схемно-валидный вывод** через constrained decoding. Это и есть фундамент паттерна «template id + parameters».

- **OpenAI Structured Outputs** (GA с авг 2024): `response_format: json_schema` со `strict: true`, и tool calling со `strict`. Под капотом — грамматика/FSM маскирует токены, ломающие схему. Заявлено **100% соответствие схеме** против **<40%** у старых моделей на одном промптинге; причём даже после дообучения модель давала ~93%, а оставшийся разрыв до 100% дал именно детерминированный слой constrained sampling. ([OpenAI launch](https://openai.com/index/introducing-structured-outputs-in-the-api/) ⚠️ страница вернула 403 фетчеру, цифры подтверждены вторично через [Humanloop](https://humanloop.com/blog/structured-outputs); [OpenAI structured-outputs guide](https://developers.openai.com/api/docs/guides/structured-outputs))
- **Anthropic Structured Outputs** (beta ноя 2025, ныне GA): `output_config.format = { type: "json_schema", schema }` и `strict: true` на инструментах. Описано как **«constrained sampling с компилируемыми грамматическими артефактами»**, грамматика кэшируется на 24 ч. Поддержано на линейке Sonnet/Opus/Haiku 4.5 и новее (включая `claude-opus-4-8`, на котором мы работаем). ([Anthropic docs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs))
- **Критично — лимиты JSON-Schema у Anthropic:** грамматика **НЕ** обеспечивает числовые диапазоны (`minimum`/`maximum`/`multipleOf`), длины строк, рекурсию, внешние `$ref`. **Поддержано:** `enum`, `const`, `anyOf`, `allOf`, внутренний `$ref`, `required`, `additionalProperties:false`. Потолки на запрос: ≤20 strict-инструментов, ≤24 опц. параметров, ≤16 union-параметров.
  - **Вывод:** LLM-слой гарантирует, что ширина — *число*, а `template_id` — из *enum*, но **не** гарантирует ширину ∈ [300,1200] и кратность сетке. Это — работа нашего слоя валидации каталога.

**Почему constrained/template бьёт free-form:**
- Синтаксическая валидность гарантирована, а не «как повезёт» — исчезает целый класс «невыполнимого вывода», retry/parse-машинерия. ([Better ML](https://medium.com/better-ml/herding-llms-structured-output-with-constraints-ae157ecf5d81))
- LLM **выбирает из нашего каталога, а не изобретает геометрию**: «выбери `template_id` (enum) + заполни объявленные параметры» → нельзя произвести несуществующий тип.
- **Ключевая оговорка:** constrained decoding гарантирует *синтаксис*, но **не семантику** — валидный JSON может содержать бессмысленные значения. Наш движок ограничений каталога — единственный источник истины и финальный валидатор. ([ingramhaus](https://ingramhaus.com/schema-constrained-prompts-how-to-force-valid-json-and-structured-llm-outputs))
- **Reason-then-emit:** дать модели сначала свободный «черновик» рассуждения, затем strict tool call только для финальных параметров; форсирование схемы на самом шаге рассуждения снижает точность. ([letsdatascience](https://letsdatascience.com/blog/structured-outputs-making-llms-return-reliable-json))

**Исследовательский сигнал «LLM как конфигуратор»:** ACMG (NL → формальная CSP-модель через извлечение сущностей → генерацию ограничений → итеративную валидацию) — зеркалит наше разделение «LLM предлагает / solver валидирует». ([MDPI Appl. Sci. 15(12):6518](https://www.mdpi.com/2076-3417/15/12/6518))

**Рекомендация:** LLM эмитит через strict tool call `{ template_id (enum), parameters (схема под шаблон) }`; схема ловит форму/enum/required; наш **движок ограничений** ловит диапазоны, снап к сетке, совместимость и кросс-параметрические правила и порождает геометрию. **Готово к проду сегодня** (мы уже на `claude-opus-4-8`).

### 5.2 Генеративные картинки для каталога: depth/normal из three.js → ControlNet

Идея — отрендерить depth/normal/edge-пасс из параметрической 3D-базы, затем фотореалистично «раскрасить» по нему — это ровно то, для чего сделан ControlNet, и это **готово к проду на fal.ai**.

- **fal.ai** — serverless-хостинг быстрых image/video/3D API. ([fal.ai](https://fal.ai/))
- **`fal-ai/flux-control-lora-depth/image-to-image`** — принимает `prompt`, `image_url` (init для цвета/стиля) и **`control_lora_image_url` (карта глубины)** для переноса структуры. Буквально «depth → стилизованный рендер». ([fal API](https://fal.ai/models/fal-ai/flux-control-lora-depth/image-to-image/api))
- **FLUX.2** (BFL, конец 2025) — до 4 MP, мульти-референс до 10 изображений, силён в продуктовых шотах. ⚠️ Native depth-control у FLUX.2 на fal **не подтверждён** — использовать подтверждённые FLUX.1[dev]/[pro] depth-эндпоинты. ([fal.ai/flux-2](https://fal.ai/flux-2))
- **ControlNet** (Zhang et al., 2023): depth/canny/lineart/normal/MLSD. Для нас полезнее всего **depth** (общая структура), **canny/lineart** (резкие края дверей/ручек), **MLSD** (прямые линии — явно про интерьер/архитектуру). Карту глубины можно не оценивать (Depth Anything v2), а **рендерить истинный depth/normal-буфер прямо из three.js** — это точнее. ([arXiv 2302.05543](https://arxiv.org/pdf/2302.05543), [Stable Diffusion Art ControlNet](https://stable-diffusion-art.com/controlnet/))

**Почему важно:** так как геометрия полностью определяется template+parameters, можно рендерить детерминированный depth/normal/edge-пасс на каждую конфигурацию и кормить ControlNet. Диффузия рисует только материалы/свет/стилистику поверх *нашей* структуры → миниатюры геометрически верны товару, а **пресет** = фиксированный style-prompt + настройки ControlNet. Правильное разделение труда: параметрический движок владеет геометрией, диффузия — внешним видом.

⚠️ Строгого первичного кейса производителя мебели с depth-ControlNet-пайплайном для каталога не нашлось — только маркетинг вертикальных SaaS (Visualizee.ai, ReRoom, Planner 5D); «30 сек/фотореализм» считать непроверенным.

### 5.3 AI генерация параметрической мебели / реконструкция из фото и чертежей

- **Image-to-3D (готово к проду, но даёт МЕШИ, не параметрику):** TRELLIS/TRELLIS.2 (Microsoft, **MIT-лицензия**), Hunyuan3D 2.1/2.5 (Tencent), Tripo 2.5/3.0, Meshy, Rodin/Hyper3D, Stable Fast 3D/TripoSR. Многие хостятся на fal. ([3DAI Studio сравнение](https://www.3daistudio.com/blog/best-3d-model-generation-apis-2026), [Hunyuan3D-2.1](https://github.com/Tencent-Hunyuan/Hunyuan3D-2.1)). **Оговорка:** выдают неструктурированные меши (marching-cubes), не редактируемые параметрические шкафы. Полезно как **помощь в ингесте ассетов** (референс-меш из фото конкурента/скетча, декоративные пропсы), **не** как ядро конфигуратора.
- **Text/Image-to-CAD и синтез параметрических программ (research-stage, но правильная *форма* идеи):** CAD-Llama (LLM эмитит «Structured Parametric CAD Code», 84.7% text-to-CAD; [arXiv 2505.04481](https://arxiv.org/html/2505.04481v2)), CAD-GPT (AAAI 2025), FutureCAD (LLM → исполняемые CadQuery-скрипты; ⚠️ [arXiv 2603.11831](https://arxiv.org/abs/2603.11831), 2026, абстракт-онли), cadrille ([arXiv 2505.22914](https://arxiv.org/pdf/2505.22914)), image-to-CAD GIFT (⚠️ 2026). Бенчмарки: Text2CAD (NeurIPS 2024), Text2CAD-Bench.
  - **Почему важно:** вся линия CAD-LLM подтверждает нашу ставку — LLM, эмитящий *структурированную параметрическую программу*, корректнее free-form-геометрии, а исполнение/валидация бьют свободную генерацию. Но это не plug-and-play: общий машиностроительный CAD, точность ~80%, сильнейшие результаты 2026 — непроверены. **Практический вывод: брать архитектуру, не модели.** Наши рукописные шаблоны *уже и есть* та «структурированная параметрическая репрезентация», которую эти работы трудозатратно учат, — поэтому задача LLM сводится к гораздо более надёжному «заполнить параметры известного шаблона».

### 5.4 Главные выводы по AI-стратегии

- **LLM — *предлагатель параметров*, не *генератор геометрии*.** Strict tool calling / Structured Outputs → `{ template_id (enum), parameters }`. Готово к проду на `claude-opus-4-8`.
- **Constrained decoding гарантирует синтаксис, не семантику.** Наш движок ограничений остаётся источником истины и финальным валидатором (диапазоны, снап к сетке, совместимость — грамматикой Anthropic *не* обеспечиваются).
- **Reason-then-emit:** свободный черновик → strict tool call для финальных параметров.
- **Миниатюры каталога: depth/normal из three.js → ControlNet на fal** (`flux-control-lora-depth`). Параметрика владеет геометрией, диффузия — видом; пресет = style-prompt + конфиг ControlNet. Готово к проду.
- **Image-to-3D (TRELLIS/Hunyuan3D/Tripo) — зрелое, но даёт меши**, не параметрику. Только как помощь в ингесте, не ядро.
- **Text-to-CAD/CAD-LLM подтверждает парадигму, но не компонент.** Брать архитектуру, не модели. Research-stage.

## 6. Топ-8 выводов для архитектуры нашего каталога

1. **Слоистая модель, как в OFML/IDM.** Три независимых слоя — (1) параметрическая геометрия, (2) коммерческие данные (цена/SKU/опции), (3) правила/ограничения — связанные через идентичность артикула. Это подтверждено и офисным (OFML), и кухонным (IDM, родной для Häcker) стандартами.
2. **Шаблон = параметрический article, SKU генерируются, а не перечисляются.** Конфигурация сериализуется в вариант-код (как OFML variant code / IDM `FEATURE/OPTION`). Наши «пресеты» — именованные, заранее провалидированные точки в пространстве вариантов.
3. **Декомпозиция корпус / фасады / фурнитура — ортогональные слои поверх корпуса** (блюпринт — IKEA METOD). Поток: линейка/стиль фасада → шкаф → отделка → ручка. Отделку применять *к* геометрии, не плодить N×M.
4. **«Система 32 мм» — детерминированный движок размещения.** Передний ряд 37 мм, всё на решётке `37 + 32·n`, высоты фасадов `32·k − зазор`; Ø5 (системные) vs Ø8 (присадочные) — разные паттерны. Размеры — параметрические сетки (ширины шаг 100 мм; высота столешницы = корпус+цоколь+столешница; толщина 18 мм — мастер-офсет).
5. **Гибридная геометрия на вебе:** процедурный box/extrude-корпус + GLB-пресеты фурнитуры/профильных фасадов, размещаемые по параметрам и async-подменяемые за плейсхолдерами; InstancedMesh для идентичных повторов, mergeGeometries — для финализированного шкафа; цель < 100 draw calls/кадр.
6. **Производительность определяют материалы и шейдеры, а не полигоны.** Малая фиксированная палитра общих `MeshStandardMaterial`, плейсхолдер-текстуры вместо `null`, прогрев `compileAsync` перед показом — прямое лекарство от нашего `shader-compile-freeze-transitions`. Обязательный `dispose()` на пересборке; GLB — Draco+KTX2. Вырезы — `three-bvh-csg`, резать один раз на коммите. Текстуры дерева — triplanar.
7. **AI: LLM — предлагатель параметров, не геометрии.** Strict tool calling → `{ template_id, parameters }`; reason-then-emit; финальная валидация — наш движок ограничений (constrained decoding даёт синтаксис, не семантику и не числовые диапазоны).
8. **AI для контента, не для ядра:** миниатюры — depth/normal из three.js → ControlNet/FLUX на fal.ai (детерминированная геометрия → стилизованный вид, пресет = style-prompt + конфиг). Image-to-3D и Text-to-CAD — зрелое/исследовательское соответственно, но только как вспомогательный ингест, не как параметрическое ядро.

## Приложение: сводка флагов «не подтверждено»

- **Häcker + OFML/pCon** — нет данных; подтверждено CARAT + IDM(DCC). pCon.update под логином 100% исключить нельзя.
- **Häcker + KPS** — вероятно (дуальная поддержка стандартна), но без прямого источника SHD/KPS.
- **OFML:** «GoDigital/GoOFML/DataConnector», «Part V», отдельная спецификация «OAS» — не подтверждены из первичных источников.
- **Размеры:** конкретный номер DIN/EN не найден; «780 мм» (Arcadium3D) ненадёжно.
- **three.js:** поддержка `compileAsync` неравномерна по браузерам; нет встроенного triplanar; `@react-three/csg` помечен WIP; UV-поведение three-bvh-csg на булевых результатах слабо документировано.
- **AI:** цифры OpenAI (100%/<40%/93%) подтверждены вторично (страница 403); ряд arXiv 2026 (FutureCAD, GIFT, Text2CAD-Bench) — абстракт-онли; строгого первичного кейса производителя мебели с depth-ControlNet каталогом не найдено (только маркетинг).
